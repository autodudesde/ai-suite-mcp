# Claude Desktop as a local MCP client

This document covers **every step required to wire the AI Suite MCP server into Claude Desktop**.

Claude Desktop runs locally on macOS / Windows / Linux — the connection to the MCP server is made from the user's machine, **not** from Anthropic's infrastructure. The AI Suite MCP server therefore does **not** need to be publicly reachable: localhost, `*.ddev.site`, internal DNS names, and self-signed certificates all work, as long as your local OS trusts the CA.

For background on Claude Desktop and MCP refer to the official docs:

- **Claude Desktop downloads** — [claude.ai/download](https://claude.ai/download)
- **MCP user quickstart (Anthropic)** — [Quickstart for users](https://modelcontextprotocol.io/quickstart/user)
- **Model Context Protocol** — [Specification](https://modelcontextprotocol.io/), [Anthropic MCP overview](https://docs.anthropic.com/en/docs/agents-and-tools/mcp)

## Conventions used in this guide

- **`<typo3-url>`** — base URL of the TYPO3 instance that has the AI Suite MCP extension installed and is reachable **from your local machine** (e.g. `https://typo3.example.com`, `https://my-project.ddev.site`, or `http://localhost:8080`). Self-signed certificates work as long as the issuing CA is trusted by your OS.
- **`<token>`** — the bearer token you create in Step 1 (long random string, prefixed by the AI Suite backend).
- **Authentication mode:** this guide uses **static bearer tokens** generated by the AI Suite backend, which is the simplest path for a desktop client and does not require OAuth setup. If you prefer the OAuth-based remote-connector flow (same flow that Claude.ai uses), see [`claude-ai.md`](claude-ai.md) — the allowlist values and authorize endpoint are identical between Claude.ai and Claude Desktop.

## Service URLs

| Service | URL |
|---|---|
| AI Suite MCP endpoint | `<typo3-url>/aisuite-mcp` |

## Prerequisites before configuring Claude Desktop

1. **Enable the AI Suite MCP extension** — TYPO3 backend → *Admin Tools → Settings → Extension Configuration → `ai_suite_mcp`*:

   | Setting | Value |
   |---|---|
   | `enableMcp` | `1` |
   | `mcpAllowedRedirectUris` | not used by the static-token flow — leave as is |
   | `mcpAllowedOrigins` | not used by the static-token flow — leave as is |
   | `mcpAllowHttp` | `1` *only* if `<typo3-url>` is plain HTTP (e.g. local non-TLS dev). Production: keep `0` |

2. **BE user group permissions** — see [Required BE-group permissions](../README.md#required-be-group-permissions) in the main README for the full list of feature flags to grant.

3. **The TYPO3 host is reachable from your local machine** — verify in a terminal on the same machine that runs Claude Desktop:
   ```bash
   curl -sS <typo3-url>/aisuite-mcp/health
   ```
   Must respond `200`. If you use a self-signed certificate (e.g. DDEV's mkcert), make sure the CA is installed in your OS trust store (`mkcert -install`).

4. **Claude Desktop is installed** and you have signed in with any plan tier — local MCP servers work on the free plan.

## Step 1 — Create an access token in the AI Suite backend

1. In the TYPO3 backend, open the **MCP** tab in the AI Suite backend module (or click the MCP icon in the AI Suite button bar).
2. Click **Create Token** — the dashboard generates a token bound to your BE user with all scopes you are entitled to (per the BE-group permissions from prerequisite #2).
3. Copy the **`claude_desktop_config.json` snippet** the dashboard returns. It looks like this:

   ```json
   {
     "mcpServers": {
       "typo3-ai-suite": {
         "url": "<typo3-url>/aisuite-mcp",
         "transport": "http",
         "headers": {
           "Authorization": "Bearer <token>"
         }
       }
     }
   }
   ```

   Token lifetime is controlled by `mcpTokenLifetimeDays` in the extension configuration (default `30`).

## Step 2 — Paste the snippet into the Claude Desktop config

1. Locate the Claude Desktop config file — create it if it does not exist:

   | OS | Path |
   |---|---|
   | macOS | `~/Library/Application Support/Claude/claude_desktop_config.json` |
   | Windows | `%APPDATA%\Claude\claude_desktop_config.json` |
   | Linux | `~/.config/claude/claude_desktop_config.json` |

2. **Merge** the snippet from Step 1 into the existing config — do not blindly overwrite if you already have other MCP servers under `mcpServers`. Example merged file:

   ```json
   {
     "mcpServers": {
       "some-other-server": { "command": "...", "args": ["..."] },
       "typo3-ai-suite": {
         "url": "<typo3-url>/aisuite-mcp",
         "transport": "http",
         "headers": {
           "Authorization": "Bearer <token>"
         }
       }
     }
   }
   ```

3. Save the file and **fully quit and restart Claude Desktop** (a window close is not enough; use the macOS Quit, Windows tray exit, or `pkill -x Claude` / equivalent).

## Step 3 — Verify the connector is loaded

1. Open a new conversation in Claude Desktop.
2. Click the **🔨 tools / connectors icon** below the chat input (the exact label is *"Search & tools"*, *"Connectors"*, or a hammer glyph depending on the build).
3. The list should include the AI Suite tools (`getServerInfo`, `getTables`, `getPageTree`, `generateMetadata`, …) under the server entry `typo3-ai-suite`.
4. If the entry is missing, the config file did not parse — see Troubleshooting.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `typo3-ai-suite` does not appear in the tools list after restarting Claude Desktop | `claude_desktop_config.json` has a JSON syntax error, or it lives in the wrong path for the OS | Validate the file with `python -m json.tool < claude_desktop_config.json`; double-check the path from Step 2 |
| Tools list shows the server but every call returns *"Authentication required"* / 401 | Token is invalid, expired, or was revoked; or the URL in the config has a TYPO3 site prefix that the MCP middleware does not match | Re-issue the token via Step 1 (old tokens are invalidated automatically). Make sure the URL is the **root** form `<typo3-url>/aisuite-mcp` — not `<typo3-url>/<site>/aisuite-mcp` |
| `SSL: CERTIFICATE_VERIFY_FAILED` / `unable to verify the first certificate` in Claude Desktop logs | The TYPO3 host uses a self-signed certificate whose CA is not in your OS trust store | For DDEV: run `mkcert -install` once on your host, then restart Claude Desktop. For other dev setups: import the CA into your OS keychain / certificate store |
| Model says *"I have no access to MCP"* even though the tools list shows the server | The conversation does not have the AI Suite tools enabled in the per-chat tool picker | Open the **🔨 tools / connectors** menu under the chat input and toggle the AI Suite tools on |

For state-gate, RateLimiter DI-cache, Apache `Authorization`-header strip and empty-tools-list issues that affect any client, see [Common troubleshooting](../README.md#common-troubleshooting) in the main README.

## Live logs while debugging

- **Claude Desktop** — application logs land in:

  | OS | Path |
  |---|---|
  | macOS | `~/Library/Logs/Claude/` (look for `mcp.log` and `mcp-server-typo3-ai-suite.log`) |
  | Windows | `%APPDATA%\Claude\logs\` |
  | Linux | `~/.config/claude/logs/` |

- **AI Suite MCP** — TYPO3 exception pages are returned directly in the HTTP response. Stack traces also land in `/var/log/` where applicable.
- **Webserver access log** — watch for `404` on `/aisuite-mcp` (site-prefix problem) or repeated `401` on `/aisuite-mcp` (Authorization-header problem).

## Persistence notes

- The bearer token is stored in the AI Suite MCP database table `tx_aisuite_oauth_tokens` (the static-token flow reuses the same token store as the OAuth flow). Lifetime is controlled by `mcpTokenLifetimeDays` in the extension configuration (default `30`).
- The Claude Desktop side stores nothing beyond the config file you wrote in Step 2 — there is no separate token cache.
- Revoking access can be done from either side — in Claude Desktop by removing the entry from `claude_desktop_config.json` (and restarting), or in the TYPO3 backend via the **MCP dashboard → Revoke Token**.
